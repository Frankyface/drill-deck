## #9 Multiple categories per drill
- **Success:** a drill can carry several categories; the library filter matches
  a drill if ANY of its categories is selected; the drill form multi-selects
  categories; existing single-category drills keep working (their one category
  becomes their category set).
- **Model:** new `drill_category_links(drill_id, category_id)` m2m. Keep
  `drills.category_id` as the *primary* (first) for card display + back-compat;
  backfill it into the link table. RLS mirrors other drill-child tables.
- **Verify:** migration applied; backfill count == drills count; RLS probe
  (member reads links of a visible drill, non-owner can't insert). Unit tests:
  filter matches on any category; DrillForm round-trips a multi-category set.
  Browser: select 2 categories, save, reopen — both shown; filter by the second
  finds it.

## #10 Sessions without a team
- **Success:** a coach can create a "personal" session (no team) visible only
  to them; the session builder offers "Just me" alongside their teams; team
  sessions unchanged.
- **Model:** `sessions.team_id` nullable. RLS: `team_id IS NULL` → owner-only
  (select/insert/update/delete on `created_by = auth.uid()`); non-null → team
  rules (existing). `session_items`/reviews inherit via the session.
- **Verify:** migration + RLS probe (owner sees own personal session; another
  user gets 0 rows). Unit test: session total/phase logic unchanged for
  team_id null. Browser: create a personal session, add drills, run it.

## #8 Keyboard doesn't hide bottom form fields
- **Success:** on a phone, focusing any field in the new-drill / new-session /
  auth forms keeps that field visible above the keyboard; the form scrolls.
- **Impl:** `KeyboardAvoidingView` (iOS padding / Android height) around the
  scrolling `Screen`, `keyboardShouldPersistTaps`, `automaticallyAdjustKeyboardInsets`,
  and scroll-to-focused on the active input.
- **Verify:** tsc/jest green; browser render unchanged. `[device]` real keyboard
  test on Cam's phone (harness has no soft keyboard).

## #1 Click-to-edit elements (number, colour, team, delete)
- **Success:** tapping a player/cone/ball/tackle-gear in the editor opens an
  edit panel: change a player's number (label) and team/colour, delete any
  element, (players) toggle a tackle shield. Long-press delete still works.
- **Verify:** unit tests for the immutable edit ops (setLabel, setTeam, remove).

  selected (selection via testID-driven state, since synthetic taps are
  unreliable). `[device]` tap-to-select + edits.

## #2 Live draggable pass receive-point (Real Rugby parity)
- **Success:** when a pass exists, a live "lead line" is drawn from the passer's
  release point to the computed receive point (where the receiver will be);
  that receive point is a draggable handle — dragging it overrides the solver
  and the override is saved on the pass; clearing reverts to auto-solve.
- **Model:** `Pass.receiveOverride?: Vec2 | null` in scene schema. Playback +
  editor use the override when present, else the interception solver.
- **Verify:** unit tests: solver used when no override; override respected when
  set; override serialized/migrated. Browser: lead line renders; handle present.
  `[device]` drag the handle.

## #3 Segment speed bar (per-segment jog/run/sprint/HOLD)
- **Success:** with a run selected, a bar at the bottom shows the run split into
  segments; each segment's speed cycles jog/run/sprint/**hold** (hold = a
  timed stop, feet planted); segment boundaries adjustable. Real Rugby's
  speed-segment model, our location.
- **Model:** `Run.segments?: { fromFrac; toFrac; speed: 'walk'|'jog'|'run'|'sprint'|'hold'; holdMs? }[]`
  (absent = single `speed` as today). Playback: piecewise kinematics; a `hold`
  segment freezes the runner for `holdMs` (or until its share of path-time),
  keeping smoothstep within moving segments.
- **Verify:** unit tests: single-speed back-compat; a hold segment pauses then
  resumes at the right place/time; total duration = sum of segment times.
  Browser: the segment bar renders for a selected run; speed chips cycle incl.
  hold. `[device]` drag boundaries + watch a hold.

## #4 Posts + 5m/15m lines on the full field
- **Success:** the `full` pitch background gains goalposts at both try lines and
  the 5m and 15m lines (dashed, both sides) in addition to the existing 22/10/
  halfway lines.
- **Verify:** browser: on a full-pitch scene the SVG contains the added
  posts/lines (element count + positions). Visual check via screenshot.

## #5 Free zoom / crop (start full-field, zoom & pan)
- **Success:** the editor canvas starts showing the whole pitch and supports
  pinch-zoom + pan to focus an area; a "fit" button resets; the chosen viewport
  is remembered per diagram (stored) so it reopens where you left it. Drawing/
  dragging still map to true pitch metres at any zoom.
- **Model:** `SceneV5.viewport?: { cx; cy; zoom }` (pitch-metre centre + scale).
  All px↔metre conversions factor in viewport; gestures stay worklet-safe
  (raw px via runOnJS, per the build-3 rule).
- **Verify:** unit tests: viewport metre↔px transforms invert; drawing at zoom
  lands at correct metres. Browser: fit/zoom buttons change the transform;
  tokens stay at correct relative positions. `[device]` pinch + pan.

## #6 Tackle shields (on players) + tackle towers (placeable)
- **Success:** a player can be given a tackle shield (rendered as a held pad);
  tackle towers are a new placeable element (like cones) with their own icon.
  Both are taggable equipment conceptually but here they're diagram elements.
- **Model:** `PlayerElement.gear?: 'shield' | null`; new element type
  `'tackle-tower'`. Rendering + palette + edit-panel toggle (shield).
- **Verify:** unit tests: add tower, toggle shield (immutable). Browser: palette
  has "+ Tackle tower"; a shield renders on a player; both survive save/reload.
